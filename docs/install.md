# Installation Guide

> Full end-to-end guide. Run on a fresh Ubuntu machine.

## Requirements

| Requirement | Minimum | Recommended |
|-------------|---------|-------------|
| OS | Ubuntu 20.04 | Ubuntu 22.04 or 24.04 |
| RAM | 4 GB | 8 GB |
| Disk | 10 GB free | 20 GB free |
| Architecture | x86\_64 | x86\_64 |
| Internet | Required (install-time only) | — |

## Quick Install

```bash
# Clone the repo
git clone https://github.com/ashishbhatt93/ProdOps-chronicles.git
cd prodops-chronicles

# Run the install script (requires sudo)
chmod +x scripts/install.sh
sudo ./scripts/install.sh
```

The install script runs these steps in order:

1. Checks OS (Ubuntu 20.04 / 22.04 / 24.04), architecture (x86\_64), RAM, disk space, and internet connectivity
2. Walks you through a short Linux primer covering the commands the script uses
3. Installs core dependencies: `curl`, `wget`, `git`, `ca-certificates`
4. Installs Docker (CE) and Docker Compose plugin from the official Docker apt repository
5. Adds your user to the `docker` group so you can run Docker without `sudo`
6. Creates system users: `postgres` (UID 999) and `prodops` (UID 1000)
7. Creates the directory structure under `/opt/prodops/`
8. Generates `base_configs.yaml` — all paths, versions, modules, and difficulty level
9. Generates `values.yaml` — module unlock order reference

## After Installation

```bash
# Log out and back in (or run this) to use Docker without sudo
newgrp docker

# Start ProdOps
prodops start

# Enable your first module
prodops module enable linux-cli

# Start learning
prodops lesson next
```

## Configuration

All runtime configuration lives in `base_configs.yaml` (generated by the installer, gitignored). The two most commonly changed settings are:

### Changing difficulty

Edit `base_configs.yaml` and restart the backend:

```yaml
difficulty: d1   # DevOps Engineer — all hints, standard pressure
# difficulty: d2 # Senior DevOps Engineer — 2 hints, tighter windows
# difficulty: d3 # DevOps Team Lead — 1 hint, strictest scoring
```

Then restart:

```bash
prodops stop && prodops start
```

### Enabling modules

By default, only the 6 v1.0 beginner modules are enabled. To enable a v2.0 module (requires `runtime: k3s`):

```yaml
modules:
  terraform:
    enabled: true
    min_difficulty: d1
    port: null
```

Restart the backend after any change to `base_configs.yaml`.

### Changing the base path

If you need data on a different volume, change `storage.base_path` before the first run. All sub-paths (`pgdata_path`, `modules_path`, `sync_path`, etc.) derive from `base_path` automatically unless you override them individually:

```yaml
storage:
  base_path: /mnt/data/prodops   # everything goes here
  pgdata_path: /mnt/ssd/pgdata   # optional: override just pgdata to a faster disk
```

## Directory Structure

After installation, `/opt/prodops/` contains:

```
/opt/prodops/
  pgdata/       Postgres data directory (owned by postgres UID 999)
  modules/      Module state (per-module subdirectories)
  sync/         prodops sync export directory (pushed to your git remote)
  logs/         Backend logs
  backend/      Backend binary state
  docker-compose.yml
  base_configs.yaml
```

## Troubleshooting

### Docker permission denied

```bash
sudo usermod -aG docker $USER
newgrp docker
```

### UID conflict during install

If the installer reports that UID 999 or 1000 is already taken:

1. Open `base_configs.example.yaml`
2. Change `postgres_uid` or `prodops_uid` to an available UID
3. Delete `base_configs.yaml` if it was already generated
4. Re-run: `sudo ./scripts/install.sh`

### Re-running the installer

The installer is idempotent — safe to re-run at any time. It skips steps that are already complete. `base_configs.yaml` and `values.yaml` are not overwritten if they already exist. Delete them first if you want a clean regeneration:

```bash
rm base_configs.yaml values.yaml
sudo ./scripts/install.sh
```

### Backend won't start

The backend reads `base_configs.yaml` at startup. If the file is missing or has a syntax error, it exits immediately. Check:

```bash
# View backend logs
prodops status

# Validate the config file manually
cat /opt/prodops/base_configs.yaml
```

Common issues:
- `storage.base_path` is missing or empty — required field
- `runtime` is not `compose` or `k3s`
- `difficulty` is not `d1`, `d2`, or `d3`
- A module `min_difficulty` value is misspelled

### Module not appearing after enabling in base_configs

1. Check `enabled: true` is set for the module in `base_configs.yaml`
2. Check your `difficulty` level meets the module's `min_difficulty`
3. Restart the backend: `prodops stop && prodops start`
4. Check the module is unlocked in the DB: `prodops module list`
   — a module can be enabled in config but still locked behind a prerequisite
